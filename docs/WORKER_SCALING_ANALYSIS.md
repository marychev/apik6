## 2. Почему увеличение числа воркеров не помогло

### 2.1. Модель воркеров uvicorn

Когда мы запускаем `uvicorn --workers N`, происходит следующее:

```
uvicorn (master process)
  ├── fork() → Worker 1  (свой PID, своя память, свой GIL)
  ├── fork() → Worker 2
  ├── fork() → Worker 3
  ...
  └── fork() → Worker N
```

Каждый воркер — это **отдельный процесс ОС** (не поток!). После `fork()` каждый процесс получает **копию** всей памяти родительского процесса. Это значит:

- Глобальная переменная `_producer = None` в `kafka_app/producer.py` копируется в каждый воркер
- При первом запросе каждый воркер создаёт **свой собственный** `KafkaProducer`
- **N воркеров = N независимых TCP-соединений** к одному и тому же Kafka-брокеру

```
Worker 1  →  KafkaProducer #1  ─┐
Worker 2  →  KafkaProducer #2  ─┤
Worker 3  →  KafkaProducer #3  ─┼──▶  Kafka broker (1 partition)
...                             │
Worker N  →  KafkaProducer #N  ─┘
```

> **Ключевой вывод:** увеличение воркеров не "распараллеливает" Kafka — оно создаёт больше соединений к одному и тому же брокеру, конкурирующих за одну и ту же партицию.

### 2.2. GIL — почему это НЕ причина

При обсуждении производительности Python часто упоминают GIL (Global Interpreter Lock). Разберём, почему здесь это **нерелевантно**.

**Что такое GIL:**
- GIL — это мьютекс в CPython, который позволяет исполняться только одному потоку одновременно внутри одного процесса
- GIL ограничивает **многопоточность** (`threading`), но не **многопроцессность** (`multiprocessing`)

**Почему GIL не играет роли:**
- Uvicorn с `--workers N` создаёт N **процессов**, не потоков
- Каждый процесс имеет **свой собственный GIL**
- Процессы работают **параллельно** на разных ядрах CPU без ограничений GIL

```
Многопоточность (threading):       Многопроцессность (workers):
┌─────────────────────┐            ┌──────────┐  ┌──────────┐
│  Процесс            │            │ Worker 1 │  │ Worker 2 │
│  ┌──────┐ ┌──────┐  │            │ свой GIL │  │ свой GIL │
│  │Поток1│ │Поток2│  │            │ параллель│  │ параллель│
│  └──┬───┘ └──┬───┘  │            └──────────┘  └──────────┘
│     └─ GIL ──┘      │                 ▲              ▲
│   (по очереди!)      │                 └── независимы ┘
└─────────────────────┘
```

> **Если кто-то скажет "Python медленный из-за GIL" — это не про uvicorn workers.** GIL был бы проблемой, если бы мы использовали `threading` внутри одного процесса. С `--workers` каждый процесс — полноценная параллельная единица.


**Что это означает для ресурсов:**

| Компонент | Потребление CPU | Потребление RAM |
|-----------|----------------|-----------------|
| API (N workers) | N × 0.5–1 ядро | N × 50–80 MB |
| Kafka (JVM) | 1–2 ядра | 256–512 MB |
| Zookeeper (JVM) | 0.3–0.5 ядра | 128–256 MB |

На типичной dev-машине с **4 физическими ядрами (8 логических потоков)**:

```
Доступно для API = Всего ядер − Инфраструктура

                 = ~4 эффективных ядра для воркеров
```

> **Ключевой вывод:** на dev-машине с Docker на WSL2 у API-воркеров есть доступ примерно к 3–5 логическим ядрам, а не ко всем 8. Это кардинально меняет расчёт оптимального числа воркеров.

### 2.4. Context switching (контекстное переключение)

Когда число процессов превышает число доступных ядер CPU, операционная система вынуждена **переключаться** между ними.

**Что происходит при контекстном переключении:**

```
Ядро CPU #1 во времени:

    Worker 1          Worker 3         Worker 1
  ├───────────┤ CS ├───────────┤ CS ├───────────┤
              ↑                ↑
        context switch    context switch
          (~1-10 мкс)      (~1-10 мкс)
        + инвалидация     + инвалидация
          L1/L2 кеша       L1/L2 кеша
          (~50-100 мкс      (~50-100 мкс
           "прогрев")        "прогрев")
```

**Стоимость переключения:**

| Элемент | Стоимость | Описание |
|---------|-----------|----------|
| Прямая стоимость | 1–10 мкс | Сохранение/восстановление регистров CPU |
| Инвалидация L1 кеша | 10–50 мкс | Новый процесс загружает свои данные |
| Инвалидация L2/L3 кеша | 50–200 мкс | При конкуренции многих процессов |
| TLB flush | 10–100 мкс | Пересчёт адресов виртуальной памяти |

**Пример для нашей системы:**

Общее число процессов на машине при 8 воркерах:
- 8 × uvicorn workers
- 1 × uvicorn master
- 1 × Kafka broker (JVM, десятки потоков)
- 1 × Zookeeper (JVM)
- 1 × ClickHouse (многопоточный)
- 1 × Consumer
- WSL2 системные процессы
- **≈ 15–20+ активных процессов/потоков на ~8 логических ядер**

При 100 VU нагрузки **каждый воркер постоянно занят** — нет idle-времени, чтобы переключение было "бесплатным". Ядро буквально вырывает CPU у одного воркера и отдаёт другому.

> **Ключевой вывод:** при 8 воркерах на ~8 логических ядер (из которых ~4 заняты инфраструктурой) происходит постоянное переключение. Каждое переключение добавляет 50–200 мкс задержки. Это накапливается в **хвосте распределения** (p95), именно там, где у нас threshold.

### 2.5. Конкуренция за Kafka брокер (producer contention)

Это **главная причина**, почему увеличение воркеров не даёт линейного роста.

**Как работает `producer.flush()`:**

```python
# kafka_app/user_producer.py
def send_users_batch(users: list[UserResponse]) -> int:
    producer = get_producer()
    for user in users:
        producer.send(KAFKA_TOPIC_USERS, key=user.id, value=user.model_dump())
    producer.flush()  # ← БЛОКИРУЕТ до подтверждения от брокера
    return len(users)
```

`producer.flush()` — это **синхронный вызов**, который:
1. Ждёт, пока все сообщения в буфере будут отправлены по сети
2. Ждёт подтверждения (ACK) от Kafka-брокера
3. Только после этого возвращает управление

**Что происходит при 100 VU и N воркерах:**

```
VU 1 ──▶ Worker 1 ──▶ flush() ──┐
VU 2 ──▶ Worker 2 ──▶ flush() ──┤
VU 3 ──▶ Worker 1 ──▶ flush() ──┤    Kafka Broker
VU 4 ──▶ Worker 3 ──▶ flush() ──┼──▶ Partition 0 (ОДНА!)
VU 5 ──▶ Worker 2 ──▶ flush() ──┤    ↓
...                              │    Обрабатывает
VU 100 ─▶ Worker N ──▶ flush() ─┘    ПОСЛЕДОВАТЕЛЬНО
```

**Kafka partition — это единица последовательности.** Все записи в одну партицию обрабатываются **строго по очереди** на уровне брокера. Это гарантия порядка в Kafka.

Это означает:
- Неважно, 4 или 8 воркеров отправляют `flush()` — брокер обрабатывает запросы к одной партиции **последовательно**
- Больше воркеров = больше одновременных `flush()` = больше ожидания в очереди брокера
- Каждый `flush()` должен дождаться, пока предыдущие `flush()` от других воркеров завершатся

**Аналогия:** представьте кассу в магазине (= Kafka partition). Неважно, пришли покупатели из 4 или 8 дверей — касса одна, очередь одна, пропускная способность одна.

> **Ключевой вывод:** `producer.flush()` + 1 партиция = фиксированная пропускная способность. Добавление воркеров не увеличивает пропускную способность Kafka, но добавляет накладные расходы (память, соединения, переключения).

### 2.6. Закон Амдала (Amdahl's Law)

Закон Амдала — фундаментальный закон параллельных вычислений, объясняющий пределы масштабирования.

**Формула:**

```
                    1
Speedup(N) = ─────────────────
              S + (1 - S) / N

S = доля последовательного (непараллелизуемого) кода
N = число параллельных воркеров
```

**Применим к нашей системе:**

Разложим время запроса на компоненты:

| Компонент | Время | Параллелизуется? |
|-----------|-------|------------------|
| `prepare_user()` | ~0.1 мс | ✓ Да (CPU, каждый воркер независимо) |
| `producer.send()` | ~0.5 мс | ✓ Частично (буферизация локальная) |
| `producer.flush()` | ~200 мс* | ✗ **Нет** (ждёт одну партицию) |
| Сетевой overhead | ~10–20 мс | ✗ Нет (одно соединение к брокеру) |

*\*под нагрузкой 100 VU; без нагрузки ~1–5 мс*

Последовательная доля: **S ≈ 200 / 220 ≈ 0.91** (91%!)

```
                   1
Speedup(4) = ───────────────── = 1.08  (+8%)
              0.91 + 0.09/4

                   1
Speedup(8) = ───────────────── = 1.09  (+9%)
              0.91 + 0.09/8

                   1
Speedup(∞) = ───────────────── = 1.10  (+10%)
              0.91 + 0
```

**Визуализация:**

```
Speedup
  ▲
  │
1.10 ┤ · · · · · · · · · · · · · · · · · · · ·  ← теоретический максимум
  │                                                (при S=0.91)
1.08 ┤          ●───────●───────●───────●──────
  │        ╱
1.04 ┤      ╱
  │    ╱
1.00 ┤──●
  │
  └──┬──┬──┬───┬───┬───┬───┬───┬──▶ Workers
     1  2  4   6   8  12  16  32

  Реальность ещё хуже — overhead от лишних
  воркеров превращает кривую в колокол:

Speedup
  ▲
1.08 ┤       ●
  │      ╱ ╲
1.04 ┤    ╱    ╲
  │  ╱      ╲
1.00 ┤●         ╲
  │            ╲
0.96 ┤             ●  ← overhead > выигрыш
  │
  └──┬──┬──┬───┬──▶ Workers
     1  4  6   8
```

> **Ключевой вывод:** когда 91% времени запроса — это ожидание Kafka (непараллелизуемая часть), даже бесконечное число воркеров даст максимум +10%. А реальные накладные расходы (context switching, память) превращают выигрыш в проигрыш.

### 2.7. Закон Литтла (Little's Law)

Закон Литтла — одна из ключевых теорем теории массового обслуживания.

**Формула:**

```
L = λ × W

L = число одновременных запросов в системе (concurrency)
λ = пропускная способность (throughput, req/s)
W = среднее время обработки запроса (latency)
```

**Применяем к нашему тесту:**

Из `k6/res2.md`:
- L = 100 (VU — зафиксировано тестом)
- W = 218.7 мс = 0.2187 с (avg duration)
- λ = L / W = 100 / 0.2187 = **457 req/s** ✓ (совпадает с наблюдаемыми 456 req/s)

**Что меняет увеличение воркеров?**

| Параметр | Зависит от воркеров? | Почему |
|----------|---------------------|--------|
| L (VU) | Нет | Фиксировано в k6 (= 100) |
| W (latency) | Почти нет | Определяется Kafka flush(), не CPU |
| λ (throughput) | Почти нет | λ = L/W, оба не меняются |

Если `W` не уменьшается (потому что flush() — bottleneck), то `λ` не растёт, сколько бы воркеров мы ни добавили.

**Когда воркеры помогают по Литтлу:**

Воркеры уменьшают `W` только если **текущие воркеры перегружены CPU** (очередь на обработку). Тогда добавление воркеров разгружает очередь → `W` падает → `λ` растёт. Но в нашем случае CPU-часть (`prepare_user`) = 0.1 мс из 220 мс. CPU не перегружен — перегружен I/O (Kafka).

> **Ключевой вывод:** закон Литтла показывает, что при фиксированных 100 VU пропускная способность растёт только если падает latency. Latency определяется Kafka flush(), а не числом воркеров.

### 2.8. Давление на память (memory pressure)

Каждый дополнительный воркер — это не только +1 процесс CPU, но и значительный объём памяти.

**Потребление памяти на воркер:**

| Компонент | Размер | Описание |
|-----------|--------|----------|
| Python-интерпретатор | ~25–30 MB | Базовый CPython runtime |
| FastAPI + зависимости | ~15–20 MB | Loaded modules |
| KafkaProducer | ~32 MB | Буфер по умолчанию (`buffer_memory`) |
| Pydantic модели | ~5 MB | Схемы + кеш валидации |
| **Итого на воркер** | **~80 MB** | |

**Суммарное потребление (все контейнеры):**

| Конфигурация | API workers | API RAM | Kafka + ZK + CH | Итого |
|-------------|-------------|---------|-----------------|-------|
| 1 worker    | 80 MB       | 80 MB   | ~800 MB         | ~900 MB |
| 4 workers   | 320 MB      | 320 MB  | ~800 MB         | ~1.1 GB |
| 6 workers   | 480 MB      | 480 MB  | ~800 MB         | ~1.3 GB |
| 8 workers   | 640 MB      | 640 MB  | ~800 MB         | ~1.4 GB |

WSL2 по умолчанию ограничивает память (~50–80% от RAM хоста). На машине с 16 GB RAM лимит WSL2 может быть ~8 GB, из которых значительная часть уходит на саму виртуальную машину.

При приближении к лимиту:
- Начинается **swap** (сброс на диск) — задержки в десятки миллисекунд
- **Page faults** увеличивают tail latency (p95/p99)
- Возможен **OOM kill** отдельных контейнеров

> **Ключевой вывод:** 8 воркеров потребляют на 560 MB больше, чем 1 воркер. На dev-машине под WSL2 это может быть разницей между "всё в RAM" и "начался swap".

### 2.9. Почему именно 6 воркеров прошло, а 4 и 8 — нет

Это самый контринтуитивный результат. Разберём гипотезу.

**Предположение о машине:** 4 физических ядра / 8 логических потоков (Hyper-Threading).

**Модель "перекрытия CPU и I/O" (CPU-I/O overlap):**

Воркер в процессе обработки запроса проходит два этапа:
1. **CPU-фаза:** `prepare_user()` + `producer.send()` — нужен CPU
2. **I/O-фаза:** `producer.flush()` — CPU не нужен, ждёт сеть

```
Воркер 1: [CPU][──────── I/O wait ────────]
Воркер 2:      [CPU][──────── I/O wait ────────]
Воркер 3:           [CPU][──────── I/O wait ────────]
                ↑
       Пока один ждёт flush(), другие используют CPU
```

Оптимальное число воркеров — когда CPU-фазы разных воркеров заполняют время I/O-фаз без конкуренции. Это зависит от соотношения `t_cpu / t_io`:

```
Оптимальные воркеры ≈ 1 + (t_io / t_cpu)

При t_cpu = 0.5 мс, t_io = 200 мс:
= 1 + 200/0.5 = 401  ← теоретически

Но ограничено доступными ядрами:
= min(401, доступные_ядра) = min(401, ~5) ≈ 5-6
```

**Почему 4 — мало:**
- 4 воркера на ~5 доступных ядер — не все ядра загружены
- Пропускная способность ограничена: иногда все 4 воркера одновременно в flush(), и CPU простаивает
- Недостаточный overlap: новые запросы ждут в очереди uvicorn

**Почему 6 — оптимально:**
- 6 воркеров на ~5 ядер — лёгкий overcommit
- Всегда есть воркер, готовый принять запрос, пока другие ждут flush()
- Минимальный context switching (6 процессов на 5 ядер — переключения редки)
- Overhead ещё не перевешивает выигрыш от overlap

**Почему 8 — слишком много:**
- 8 воркеров на ~5 ядер — постоянный context switching
- 8 × KafkaProducer = повышенная конкуренция за Kafka broker
- На ~160 MB больше RAM, чем при 6 воркерах
- Overhead от переключений добавляет ~50–200 мкс к каждому запросу
- При p95=~300 мс разница в 45 мс (300→345) — это ~15%, что вполне объяснимо context switching

**Почему 1 — тоже не прошёл:**
- Один воркер обрабатывает запросы **строго по очереди**
- При 100 VU запросы выстраиваются в очередь
- Пока воркер ждёт flush() (~200 мс), все 99 VU стоят
- Хвост очереди (p95) получает огромную задержку

**Визуализация результатов:**

```
p95 latency (мс)
  ▲
400 ┤  ●                          ●
    │   ╲                       ╱
350 ┤    ╲        ●            ╱
    │     ╲     ╱   ╲        ╱
300 ┤ ─ ─ ╲─ ─╱─ ─ ─╲─ ─ ─╱─ ─ ─ ← threshold (300ms)
    │      ╲╱        ╲  ╱
250 ┤                  ●
    │                  6
    └──┬──┬──┬──┬──┬──┬──┬──┬──▶ Workers
       1  2  3  4  5  6  7  8
```

> **Ключевой вывод:** 6 воркеров — это "sweet spot" для конкретной машины, где overlap CPU/I/O максимален, а overhead от context switching ещё не критичен. Разница между "прошёл" и "не прошёл" составляет ~15% — это зона влияния системных факторов (переключения, кеш, память).

## 3. Как определить оптимальное число воркеров

### 3.1. Классические формулы

Существует несколько общепринятых формул для расчёта числа воркеров:

**Для CPU-bound приложений** (математика, рендеринг, компиляция):
```
workers = CPU_cores
```
Логика: каждое ядро занято вычислениями на 100%, больше воркеров = только overhead.

**Для I/O-bound приложений** (HTTP-запросы, БД, файлы):
```
workers = (2 × CPU_cores) + 1
```
Логика: пока один воркер ждёт I/O, другой использует CPU. Множитель 2 — эмпирический запас. Это **рекомендация Gunicorn** (WSGI-сервер, на котором основан multiprocessing в uvicorn).

**Для смешанной нагрузки:**
```
workers = CPU_cores × (1 + t_wait / t_compute)

t_wait    = среднее время I/O-ожидания на запрос
t_compute = среднее время CPU-работы на запрос
```

**Пример для нашей системы:**
```
CPU_cores (эффективные) ≈ 5
t_wait = 200 мс (flush)
t_compute = 0.5 мс (prepare_user + send)

workers = 5 × (1 + 200/0.5) = 5 × 401 = 2005  ← абсурд!
```

Формула даёт абсурдно большое число, потому что **не учитывает разделяемый ресурс (Kafka)**. Все воркеры ждут один и тот же брокер — добавление воркеров не уменьшает `t_wait`.

**Скорректированная формула для систем с shared I/O:**
```
workers = min(
    CPU_cores × (1 + t_wait/t_compute),     // формула overlap
    Пропускная_способность_IO_ресурса × t_wait,  // лимит I/O
    Доступные_ядра                            // лимит CPU
)
```

> **Ключевой вывод:** классические формулы — хорошая стартовая точка, но они работают для **независимого I/O** (каждый запрос ходит в свой ресурс). Когда есть shared bottleneck (одна Kafka partition), формулы дают завышенную оценку.

---

### 3.2. Определение характера нагрузки (CPU vs I/O)

Прежде чем выбирать формулу, нужно **измерить**, что именно занимает время в запросе.

**Метод 1: профилирование кода**

```python
import time

@router.post("/users/batch/{n}")
def batch(n: int = 1):
    t0 = time.perf_counter()
    users = [prepare_user(...) for i in range(1, n + 1)]
    t1 = time.perf_counter()
    
    sent = send_users_batch(users)
    t2 = time.perf_counter()
    
    print(f"CPU: {(t1-t0)*1000:.1f}ms, I/O: {(t2-t1)*1000:.1f}ms")
    return {"sent": sent}
```

**Метод 2: `docker stats` во время нагрузки**

```bash
# В одном терминале — нагрузка:
k6 run --vus 100 --duration 1m k6/k6.js

# В другом — мониторинг:
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}"
```

Интерпретация `docker stats`:

| CPU% | Значение |
|------|----------|
| < 100% на API | I/O-bound: CPU простаивает, ждёт Kafka |
| ≈ N × 100% | CPU-bound: все воркеры загружены |
| > 100% но < N × 100% | Смешанная нагрузка |

**Метод 3: проверка I/O-ожидания**

```bash
# Внутри контейнера (или WSL2):
cat /proc/<pid>/status | grep voluntary_ctxt_switches
```

Большое число `voluntary_ctxt_switches` = процесс часто засыпает на I/O.

### 3.3. Методология поиска оптимума (пошагово)

Не существует универсальной формулы — оптимум нужно **находить экспериментально**. Вот методология:

**Шаг 1: Определить доступные ядра**
```bash
# В WSL2:
nproc                    # логические ядра
lscpu | grep "Core(s)"  # физические ядра

# Проверить лимиты WSL2:
cat /proc/cpuinfo | grep processor | wc -l
```

**Шаг 2: Baseline (1 воркер)**
```bash
# Dockerfile: --workers 1
k6 run --vus 100 --duration 1m k6/k6.js
# Записать: RPS, p50, p95, p99, errors
```

**Шаг 3: CPU-bound оценка (workers = cores)**
```bash
# Dockerfile: --workers <nproc>
# Повторить тест, записать метрики
```

**Шаг 4: I/O-bound оценка (workers = 2 × cores + 1)**
```bash
# Dockerfile: --workers <2*nproc+1>
# Повторить тест, записать метрики
```

**Шаг 5: Бинарный поиск**
- Если шаг 3 лучше шага 4 → тестировать между 1 и cores
- Если шаг 4 лучше шага 3 → тестировать между cores и 2×cores+1
- Сужать диапазон по 1 воркеру

**Шаг 6: Валидация**
- Лучший результат прогнать **3–5 раз**
- Результаты могут варьироваться на ~10% (фоновые процессы, JIT, кеш)
- Если разброс > 20% — нестабильная среда, нужно изолировать ресурсы

### 3.4. Что измерять

| Метрика | Инструмент | На что смотреть |
|---------|-----------|-----------------|
| **p95/p99 latency** | k6 | Основной критерий. Именно хвост деградирует первым |
| **Throughput (RPS)** | k6 | Должен расти с воркерами (до предела) |
| **Error rate** | k6 | Рост ошибок = система за пределами |

### 3.5. Признаки: слишком много воркеров

| Симптом | Метрика | Объяснение |

| +1 воркер = +25%+ RPS | k6 throughput | Линейный рост = ещё не предел |
| p50 = p95 | k6 latency | Нет очереди, нет хвоста |

### 3.7. Учёт инфраструктуры

Формулы считают воркеров так, будто они **одни на сервере**. В реальности рядом работают другие сервисы.

**Формула с учётом инфраструктуры:**
```
effective_cores = total_cores − reserved_cores

reserved_cores = Kafka + Zookeeper + ClickHouse + Consumer + OS/WSL2
               ≈ 1.5  +  0.5      +  1.0       +  0.5     +  0.5
               = 4.0

# На машине с 8 логическими ядрами:
effective_cores = 8 − 4 = 4

# I/O-bound формула:
workers = (2 × 4) + 1 = 9  ← верхняя граница
# CPU-bound формула:
workers = 4               ← нижняя граница

# Оптимум: между 4 и 9, ближе к нижней границе
# (потому что Kafka partition — shared bottleneck)
```

**На продакшене vs dev:**

| Фактор | Dev (Docker/WSL2) | Production (K8s/VM) |
|--------|-------------------|---------------------|
| Ядра для API | 3–5 из 8 | Выделены (CPU limits) |
| Конкуренция | Все сервисы на 1 машине | Сервисы изолированы |
| Виртуализация | Windows → WSL2 → Docker | Bare-metal / 1 слой |
| Kafka | 1 broker, 1 partition | Кластер, N partitions |
| Оптимум воркеров | ~6 (эмпирически) | Другой! Нужно перетестировать |

> **Ключевой вывод:** оптимальное число воркеров **не переносится** между средами. То, что работает на dev (6 воркеров), может быть далёким от оптимума на production.

---

### 3.8. Кривая убывающей отдачи (Diminishing Returns)

Типичная зависимость "воркеры → производительность" выглядит так:

```
Throughput (RPS)
  ▲
  │                    ●────●────●────●  ← плато (I/O bottleneck)
  │               ●╱
  │           ●╱
  │       ●╱
  │   ●╱    ← линейный рост
  │ ╱
  │●
  └──┬──┬──┬──┬──┬──┬──┬──┬──▶ Workers
     1  2  3  4  5  6  7  8

p95 Latency (мс)
  ▲
  │                          ●
  │                       ●╱    ← деградация
  │                   ●╱        (context switching,
  │                              memory pressure)
  │   ●────●────●────●   ← стабильно
  │
  └──┬──┬──┬──┬──┬──┬──┬──┬──▶ Workers
     1  2  3  4  5  6  7  8

         ↑               ↑
    "sweet spot"     "too many"
```

**Оптимальная точка** — где throughput вышел на плато, а p95 **ещё не начал расти**.

## 5. Заключение

**Почему не помогло:**
Увеличение числа uvicorn workers не привело к улучшению, потому что bottleneck системы — **синхронный `producer.flush()` к одной Kafka partition**. Это непараллелизуемая операция (по Амдалу, ~91% времени запроса). Добавление воркеров лишь увеличивало конкуренцию за один и тот же ресурс + context switching + memory pressure.

**Почему 6 сработало:**
На конкретной dev-машине 6 воркеров — это "sweet spot", где overlap между CPU и I/O фазами максимален, а overhead ещё не критичен. Разница между "прошёл" и "не прошёл" составляла ~15% (300 vs 345 мс) — в зоне влияния системных факторов.

**Как выбирать воркеры:**
Не по формуле, а экспериментально: серия тестов с мониторингом CPU/RAM/latency. Оптимум зависит от машины, инфраструктуры и характера bottleneck. Не переносится между средами.

**Правильный путь:**
Вместо подбора воркеров — устранить bottleneck: `aiokafka` (async flush) + несколько Kafka partitions. Тогда воркеры действительно смогут работать параллельно.
