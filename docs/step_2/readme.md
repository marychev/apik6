# Отчёт: пиковый RPS сервиса

- Стек: FastAPI + Kafka (1 partition) + ClickHouse 
- Среда: Docker/WSL2, i3-1005G1 (2c/4t), 8GB RAM, 4 воркера

### Потолок (без SLA)

```
VU	        RPS	    avg	            errors
100–1600	~760	136ms–2.08s	    0%
6400	    596	    10.15s	        0%
```

Потолок: ~760 RPS. Не зависит от VU — растёт только latency. Bottleneck: sync flush() к 1 Kafka-партиции.

### Пик (SLA = 2s)

```
VU	    RPS	    avg	        errors	    Зона
500	    764	    652ms	    0.04%	    комфорт
600	    671–727	822–889ms	0.25–0.81%	граница
700	    560	    1.22s	    37.7%	    коллапс
```

Пик: ~670–760 RPS при SLA 2s (до 600 VU, ошибки <1%).
Cliff effect: +100 VU (600→700) — ошибки с <1% до 38%.

### Путь роста

aiokafka (async flush) + Kafka partitions 1→4–8 → потенциально 3000–5000+ RPS

# 1. Пиковые РПС и наблюдения (4 воркера)

```sh
VUS=200  k6 run k6/rps.js
VUS=400  k6 run k6/rps.js
VUS=800  k6 run k6/rps.js
VUS=1600 k6 run k6/rps.js
```

---

## VUS=200  k6 run k6/rps.js

### Ключевое наблюдение
RPS почти не вырос (734 → 760, +3.5%), но latency ровно удвоилась (136 → 262ms). Это классическая сигнатура насыщения — мы уже за коленом кривой пропускной способности.

Проверим по закону Литтла (λ = L/W):
- 100 VU / 0.136s = 735 rps ✓
- 200 VU / 0.262s = 763 rps ✓

Значит, пропускная способность системы уже упёрлась в ~760 RPS — это фиксированный потолок (sync flush + 1 партиция, ровно как прогнозировал анализ). Добавление VU теперь только удлиняет очередь, а не ускоряет обработку.

### Что это значит для поиска пика
Ошибок 0% — но это обманчиво. Сервис не «сломан», он просто терпит очередь. Настоящий слом начнётся, когда:
- либо латенси превысит HTTP-таймаут k6 (60с)
- либо у Kafka producer переполнится буфер (BufferFull)
- либо закончится RAM в контейнерах

max уже 1.27s — хвост растёт. При VU=400 ожидаю max 3–5с, при VU=800 — первые таймауты/ошибки.

```sh
VU	    RPS	    avg	    p95	    max	    errors
100	    734	    136ms	260ms	—	    0%
200	    760	    262ms	459ms	1.27s	0%
```

## VUS=400 k6 run k6/rps.js

```sh
VU	RPS	avg	p95	max	errors
400	761	524ms	1000ms	1.94s	0%
```

### Что происходит
Пропускная способность — железный потолок ~760 RPS. Три прогона подряд, RPS стоит как вкопанный.

Проверка Литтла снова сходится: 400 / 0.524 = 763 ✓

Поведение системы — плавное очередное: FastAPI/uvicorn спокойно держат входящие соединения, все запросы в итоге обрабатываются, просто дольше. Это здоровая деградация — сервис не падает, а растягивает очередь.

## VUS=800 k6 run k6/rps.js

```sh
VU	RPS	avg	p95	max	errors
800	770	1.03s	1.77s	4.5s	0%
```

RPS стоит на месте (~760), latency удваивается с каждым удвоением VU. Это чистая теория очередей — система работает на пределе пропускной способности, каждый новый VU только удлиняет ожидание.

## VUS=1600 k6 run k6/rps.js

```sh
VU	    RPS	avg	    p95	    max	    errors
1600	759	2.08s	3.43s	6.29s	0%
```

Сервис не ломается. Вообще. Он ведёт себя как идеальная очередь — принимает всё, обрабатывает с фиксированной скоростью ~760 RPS, просто увеличивает время ожидания. Это заслуга связки sync FastAPI + Kafka producer — каждый воркер просто блокируется на flush() и терпеливо ждёт.

## VUS=6400 k6 run k6/rps.js

```sh
VU	    RPS	avg	    p95	    max	    errors
6400	596	10.15s	15.65s	37.32s	0%
```
Впервые RPS упал (760 → 596, −22%). Ошибок по-прежнему 0%, но:
- avg 10 секунд, max 37 секунд — запросы висят в очереди чудовищно долго
- RPS просел потому что k6 не успевает утилизировать все 6400 VU за 1 минуту — видно по vus: 1646 min=0 max=6400 (в среднем активных меньше 6400, система не успевает «разогнаться»)
- Тест работал 67с вместо 60с — graceful stop добивал оставшиеся запросы

Это начало коллапса — не по ошибкам, а по деградации throughput. Система захлёбывается очередью.

# 2. Нужно определить SLA. Типичные значения:

```log
Тип сервиса	            SLA на ответ
Пользовательский API	500ms – 1s
Внутренний сервис   	2–5s
Batch / async	        10–30s
```

Ваш сервис — приём данных в Kafka (внутренний). Предлагаю 2 секунды — это разумный таймаут для такого эндпоинта. При нём по нашей таблице:
```
VU=800 (avg 1s) — ок
VU=1600 (avg 2s) — на грани, ~50% не уложатся
Между ними — пик
```

```sh
SLA_TIMEOUT=2s VUS=500 k6 run k6/rps.js
SLA_TIMEOUT=2s VUS=600 k6 run k6/rps.js
SLA_TIMEOUT=2s VUS=700 k6 run k6/rps.js
SLA_TIMEOUT=2s VUS=800  k6 run k6/rps.js  ===> FAIL
```

```log
VU	Прогон	RPS	avg	p95 (success)	errors
500	1	707	704ms	1.2s	2.8% (выброс)
500	2	764	652ms	1.16s	0.04%
600	1	727	822ms	1.43s	0.25%
600	2	671	889ms	1.54s	0.81%
700	1	560	1.22s	1.6s	37.7%
```
Что видно после валидации
VU=500 первый прогон (2.8%) — подтверждённый выброс. Повторный дал 0.04%.
VU=600 — зона начала деградации (~0.25–0.81% ошибок). Это граница комфорта.
VU=700 — коллапс (37.7%). Cliff effect подтверждён.


Что мы нашли - Именно тот паттерн, который искали:
VU	errors (валидированные)	Зона
500	0.04%	комфорт
600	0.25–0.81%	граница
700	37.7%	коллапс
Пик сервиса при SLA=2s: ~670–760 RPS (до 600 VU). Между 600 и 700 VU — cliff effect, ошибки прыгают с <1% до ~38%.

```log
«Сервис стабильно держит ~670–760 RPS при SLA 2 секунды на dev-машине (i3, 2 ядра, Docker/WSL2). Пропускная способность упирается в синхронный flush к одной Kafka-партиции. Деградация резкая — cliff effect при превышении ~600 concurrent connections (ошибки прыгают с <1% до ~38% при +100 VU).»
```
