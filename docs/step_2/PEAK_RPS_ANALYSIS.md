# Поиск пикового RPS сервиса (stress testing)

**Дата:** 16.04.2026 | **Среда:** Docker (WSL2, Windows 11) | **Инструмент:** k6
**Конфигурация:** FastAPI, uvicorn --workers 4, i3-1005G1 (2 ядра / 4 потока), 8 GB RAM

## TL;DR

**Стек:** FastAPI + Kafka (1 partition) + ClickHouse | **Среда:** Docker/WSL2, i3-1005G1 (2c/4t), 8GB RAM, 4 воркера

**Потолок (без SLA):**

| VU | RPS | avg | errors |
|----|-----|-----|--------|
| 100–1600 | ~760 | 136ms–2.08s | 0% |
| 6400 | 596 | 10.15s | 0% |

Потолок: **~760 RPS.** Не зависит от VU — растёт только latency. Bottleneck: sync `flush()` к 1 Kafka-партиции.

**Пик (SLA = 2s):**

| VU | RPS | avg | errors | Зона |
|----|-----|-----|--------|------|
| 500 | 764 | 652ms | 0.04% | комфорт |
| 600 | 671–727 | 822–889ms | 0.25–0.81% | граница |
| 700 | 560 | 1.22s | 37.7% | коллапс |

Пик: **~670–760 RPS при SLA 2s** (до 600 VU, ошибки <1%).
Cliff effect: +100 VU (600->700) — ошибки с <1% до 38%.

**Путь роста:** `aiokafka` (async flush) + Kafka partitions 1->4-8 -> потенциально 3000-5000+ RPS

---

## 1. Цель

Найти **пиковый RPS** — максимальную пропускную способность сервиса, при которой процент ошибок не превышает 2–5% (SLA breach). После пика — зафиксировать точку обвала (~50% ошибок).

---

## 2. Методология

### 2.1. Инструмент

Файл `k6/rps.js` — параметризованный тест без thresholds:

```bash
VUS=400 k6 run k6/rps.js                     # без таймаута (дефолт 60s)
SLA_TIMEOUT=2s VUS=600 k6 run k6/rps.js      # с SLA-таймаутом 2s
```

### 2.2. Подход — два этапа

**Этап 1. Поиск потолка RPS (без таймаута).**
Увеличиваем VU удвоениями (100 → 200 → 400 → 800 → 1600 → 6400), наблюдаем как растёт latency при фиксированном RPS. Таймаут HTTP по умолчанию (60s) — ошибок нет, видим чистую деградацию latency.

**Этап 2. Поиск пика по SLA (таймаут 2s).**
Устанавливаем SLA = 2 секунды. Запросы, не уложившиеся в 2s, считаются ошибкой. Ищем VU, при котором ошибки достигают 2–5% (пик), и VU, при котором ошибки резко прыгают (обвал).

### 2.3. Почему SLA = 2 секунды

| Тип сервиса | Типичный SLA |
|---|---|
| Пользовательский API | 500ms – 1s |
| **Внутренний сервис (наш случай)** | **2–5s** |
| Batch / async | 10–30s |

Наш эндпоинт — приём данных в Kafka (внутренний сервис). 2 секунды — разумная верхняя граница для клиента, ожидающего подтверждение отправки.

## 3. Этап 1: потолок RPS (без таймаута)

### 3.1. Результаты

| VU   | RPS  | avg    | p95    | max     | errors | Литтла (L/W) |
|------|------|--------|--------|---------|--------|--------------|
| 100  | 734  | 136ms  | 260ms  | —       | 0%     | 735 ✓        |
| 200  | 760  | 262ms  | 459ms  | 1.27s   | 0%     | 763 ✓        |
| 400  | 761  | 524ms  | 1s     | 1.94s   | 0%     | 763 ✓        |
| 800  | 770  | 1.03s  | 1.77s  | 4.5s    | 0%     | 776 ✓        |
| 1600 | 759  | 2.08s  | 3.43s  | 6.29s   | 0%     | 769 ✓        |
| 6400 | 596  | 10.15s | 15.65s | 37.32s  | 0%     | 631*         |

*VU=6400: k6 не успевает утилизировать все VU за 1 минуту (ramp-up effect).

### 3.2. Закономерность

```
VU × 2  →  latency × 2  →  RPS = const (~760)
```

Идеальная модель очереди (M/M/c): пропускная способность зафиксирована bottleneck-ом (sync `producer.flush()` к одной Kafka-партиции), каждый дополнительный VU только удлиняет время ожидания в очереди.

Закон Литтла `λ = L / W` подтверждается на каждом прогоне с точностью ±2%.

### 3.3. Визуализация

```
RPS                                  Latency (avg)
 ▲                                    ▲
800┤  ●────●────●────●────●           │              ●  10.15s
   │                       ╲          │            ╱
700┤                        ╲         │          ╱
   │                         ●  596   │        ●  2.08s
600┤                                  │      ╱
   │                                  │    ●  1.03s
   │                                  │  ╱
   │                                  │●  524ms
   │                                  ●  262ms
   │                                 ●  136ms
   └──┬──┬──┬───┬────┬────▶ VU       └──┬──┬──┬───┬────┬────▶ VU
     100 200 400 800 1600 6400         100 200 400 800 1600 6400
```

**Потолок RPS: ~760 req/s** — не зависит от числа VU (до момента коллапса при 6400).

### 3.4. Поведение при VU=6400

RPS впервые просел (760 → 596, −22%). Причина — не ошибки (их 0%), а **перегрузка очереди**: k6 не успевает «раскрутить» 6400 VU за 1 минуту, часть VU проводит время в ожидании. Это начало коллапса по throughput, но не по ошибкам.

### 3.5. Вывод этапа 1

Сервис **не падает** даже при 6400 VU и avg latency 10 секунд. FastAPI + sync Kafka producer = идеальная очередь. Ошибки в классическом смысле (500, connection refused) не возникают — сервис просто тормозит. Для определения «пика» необходимо ввести SLA-ограничение.

## 4. Этап 2: пиковый RPS при SLA = 2s

### 4.1. Результаты

Каждая конфигурация тестировалась **2 раза** для валидации. VU=500 прогон 1 (2.8% ошибок) оказался выбросом — фоновая нагрузка WSL2/Docker на dev-машине.

| VU  | Прогон | RPS | avg    | p95 (success) | errors | Зона |
|-----|--------|-----|--------|---------------|--------|------|
| 500 | 1      | 707 | 704ms  | 1.2s          | 2.8%   | выброс |
| 500 | **2**  | **764** | **652ms** | **1.16s** | **0.04%** | **комфорт** |
| 600 | 1      | 727 | 822ms  | 1.43s         | 0.25%  | граница |
| 600 | **2**  | **671** | **889ms** | **1.54s** | **0.81%** | **граница** |
| 700 | 1      | 560 | 1.22s  | 1.6s          | 37.7%  | коллапс |
| 800 | 1      | 642 | 1.06s  | 1.37s         | 28.7%  | прерван (7.5s) |

### 4.2. Cliff effect (обрыв)

Между 600 VU и 700 VU — резкий скачок ошибок:

```
Errors %
  ▲
 40%┤                         ● 700 VU (37.7%)
    │                       ╱
 30%┤                     ╱
    │                   ╱
 20%┤                 ╱        ← резкий обвал
    │               ╱
 10%┤             ╱
    │           ╱
  3%┤ ─ ─ ● ─╱─ ─ ─ ─ ─ ─ ─ ─  ← ПИК (500 VU, 2.8%)
    │       ╱
  0%┤──●──╱
    └──┬──┬──┬──▶ VU
      600 500 700
```

Всего +100 VU (600 → 700) — ошибки прыгнули с ~0% до ~38%. Это **cliff effect** — характерное поведение систем с единственным bottleneck (Kafka partition). Пока все запросы успевают пройти за < 2s — ошибок почти нет. Как только очередь вырастает настолько, что хвост latency уходит за 2s — ошибки лавинообразно нарастают.

### 4.3. Почему обрыв, а не плавная деградация

В нашей архитектуре latency определяется **длиной очереди на flush()** к одной Kafka-партиции. Распределение latency — не нормальное, а с тяжёлым хвостом:

- При VU=600: хвост (p95=1.43s) ещё укладывается в SLA (2s), ошибки ~0%
- При VU=700: средняя latency (1.22s) приближается к SLA, хвост массово выходит за 2s
- Переход между «почти все ок» и «треть не ок» — очень узкий (~50–100 VU)

Это свойство системы с **фиксированной пропускной способностью и растущей очередью**: латенси растёт линейно с VU, но доля запросов за SLA растёт **нелинейно** (S-кривая).

## 6. Причина потолка

```
POST /users/batch/1
  ├─ prepare_user()      ~0.1 мс   (CPU, параллелизуется)
  ├─ producer.send()     ~0.5 мс   (буфер, локально)
  └─ producer.flush()    ~1–200 мс (БЛОКИРОВКА, 1 партиция)
                         ↑
                    91% времени запроса = непараллелизуемая часть
```

По закону Амдала: при S=0.91, даже ∞ воркеров дадут максимум +10% RPS. Потолок определяется не CPU, а скоростью обработки одной Kafka-партиции.

## 7. Что нужно для роста

| Действие | Ожидаемый эффект |
|----------|-----------------|
| `aiokafka` (async flush) | Воркер не блокируется → обрабатывает другие запросы во время flush |
| Kafka partitions: 1 → 4–8 | Параллельная запись, снятие contention |
| Оба вместе | Потенциально 3000–5000+ RPS на том же железе |
