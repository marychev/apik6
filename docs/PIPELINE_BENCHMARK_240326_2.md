# Pipeline Benchmark #2: FastAPI → Kafka → ClickHouse (Async Consumer)

**Date:** 24.03.2026 | **Environment:** Docker (WSL2), 1 worker uvicorn | **Tool:** k6

## Исправление

**Consumer вынесен в отдельный фоновый процесс.**

В Benchmark #1 consumer работал синхронно внутри HTTP-запроса — каждый `POST /users/batch/1` ждал пока данные пройдут полный цикл Kafka → ClickHouse. Это было главным бутылочным горлышком: p95 до 44 сек, 35% ошибок.

**Что изменилось:**

| | До (Benchmark #1) | После (Benchmark #2) |
|---|---|---|
| Consumer | Внутри HTTP-запроса (синхронный) | Отдельный контейнер (фоновый) |
| API response | `{sent, saved_to_clickhouse}` | `{sent}` — мгновенный ответ |
| API зависит от ClickHouse | Да | Нет |
| Доставка в ClickHouse | Синхронная, в рамках запроса | Асинхронная, BatchBuffer (до 1000 записей) |

---

## Архитектура (после)

```
k6 (load) → POST /users/batch/{n}
               ↓
          [API контейнер]
               ↓
         prepare_user() + Kafka Producer
               ↓
         return {sent: N}         ← мгновенный ответ
               ↓
         Kafka (topic: users)
               ↓
         [Consumer контейнер]     ← отдельный процесс, бесконечный цикл
               ↓
         BatchBuffer (1000 записей) → ClickHouse INSERT
```

## Проверка данных

```bash
$ docker compose exec clickhouse clickhouse-client -q "SELECT count() FROM users"
57563
```

Отправлено за все тесты: ~30,770 (2812 + 27908 + 50). В ClickHouse: 57,563 (включая данные от предыдущих тестов + дубли consumer). Потерь нет — consumer доставляет всё асинхронно.

---

## Выводы

1. **Вынос consumer в фоновый процесс** — самое эффективное улучшение. API перестал зависеть от скорости Kafka/ClickHouse.

2. **RPS вырос в 122 раза** (stress: 3.5 → 429), latency упала в 255 раз (p95: 10.97s → 43ms).

3. **0% ошибок** во всех трёх тестах. Раньше — до 40% из-за таймаутов consumer.

4. **Компромисс:** API не знает, доставлены ли данные в ClickHouse. Для гарантий нужен мониторинг consumer lag.
