# k6 — нагрузочные тесты

Тестируем пайплайн: `POST /users/batch/{n}` → Kafka → ClickHouse

## Запуск

```bash
make k6-throughput   # нарастающая нагрузка
make k6-stress       # стресс-тест
make k6-spike        # спайк — 50k пользователей разом
```

Перед запуском убедитесь что сервисы работают: `make ps`

## Тесты

### throughput.js — пропускная способность

Нарастающая нагрузка от 5 до 100 req/s на `batch/1`. Показывает при какой нагрузке система начинает деградировать.

| Фаза | Длительность | Нагрузка |
|------|-------------|----------|
| Разогрев | 15s | 5 → 10 req/s |
| Средняя | 15s | 10 → 30 req/s |
| Высокая | 15s | 30 → 60 req/s |
| Пиковая | 15s | 60 → 100 req/s |
| Остывание | 10s | 100 → 5 req/s |

### stress.js — стресс-тест

До 20 VU одновременно, каждый делает `batch/1000`. Проверяет потери данных при массовой записи.

| Фаза | Длительность | VUs |
|------|-------------|-----|
| Старт | 10s | 1 → 5 |
| Нагрузка | 20s | 5 → 10 |
| Тяжёлая | 20s | 10 → 20 |
| Остывание | 10s | 20 → 1 |

### spike.js — спайк

50 VU одновременно, каждый `batch/1000` = 50,000 пользователей почти разом. Что будет при резком всплеске.

## Кастомные метрики

| Метрика | Описание |
|---------|----------|
| `batch_duration` | Полное время обработки запроса (prepare → Kafka → ClickHouse) |
| `kafka_lag` | Оценка лага доставки Kafka → ClickHouse |
| `total_sent` | Сколько пользователей отправлено в Kafka |
| `total_saved` | Сколько записано в ClickHouse |
| `lost_messages` | Потерянные сообщения (sent - saved) |

## Пороги (thresholds)

- **throughput**: p95 < 5s, ошибки < 10%, лаг Kafka < 3s
- **stress**: p95 < 15s, ошибки < 20%
- **spike**: ошибки < 30%

## Проверка результатов после тестов

```bash
docker compose exec clickhouse clickhouse-client -q "SELECT count() FROM users"
```
